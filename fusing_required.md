## 3. GPS SBF output format

The Mosaic H manual defines SBF as proprietary binary blocks beginning with `$@` (`0x24 0x40`). The SBF block header is 8 bytes: sync, CRC, ID, Length. Multi-byte fields are little-endian. The CRC covers ID through the end of the block. SBF timestamps use `TOW u4` in milliseconds and `WNc u2` continuous GPS week.

| SBF block | ID | Used by code | Key fields | Units/frame/time | Compliance notes |
| --- | ---: | --- | --- | --- | --- |
| PVTGeodetic | 4007 | yes | TOW, WNc, Mode, Error, Latitude, Longitude, Height, Undulation, Vn, Ve, Vu, COG, Datum, NrSV | lat/lon radians, height ellipsoid meters, velocity North/East/Up m/s, receiver GPS time | Code converts lat/lon rad->deg, sends height as ellipsoid, converts U to Down for CV7. |
| PosCovGeodetic | 5906 | yes | TOW, Mode, Error, Cov_latlat, Cov_lonlon, Cov_hgthgt, etc. | covariance elements in `m^2`, geodetic local components | Code uses diagonal variances at offsets 16/20/24 and sends sqrt as 1-sigma N/E/U uncertainty. |
| VelCovGeodetic | 5908 | yes | TOW, Mode, Error, Cov_VnVn, Cov_VeVe, Cov_VuVu, etc. | covariance in `m^2/s^2`, N/E/U velocity | Code uses diagonal variances and sends sqrt as 1-sigma N/E/D uncertainty. |
| AttEuler | 5938 | yes | TOW, WNc, NrSV, Error, Mode, Reserved, Heading, Pitch, Roll, PitchDot, RollDot, HeadingDot | heading/pitch/roll degrees, receiver time | Code's core heading offset is `blk+20`, which matches the manual after reserved bytes. Latest output is still invalid (`ovf`), so receiver attitude config or validity is failing. |
| AttCovEuler | 5939 | yes | TOW, WNc, Reserved, Error, Cov_HeadHead, Cov_PitchPitch, Cov_RollRoll | variance in degree^2 | Code reads variance fields at offsets 16/20/24 and sends sqrt heading sigma in radians. |
| ReceiverStatus | 4014 | configured in README, not parsed for aiding | receiver status, external errors | 1 Hz | Not used for MIP aiding in current code. |
| QualityInd | 4082 | configured in README, not parsed for aiding | quality indicators 0..10 | 1 Hz | Not used for MIP aiding in current code. |
| ReceiverTime | 5914 | not currently parsed | UTC and receiver sync level | 1 Hz | Could be used to audit leap seconds/sync but not used now. |
| PPS/xPPSOffset | xPPSOffset exists in manual | not parsed | PPS timing offsets | time sync | Teensy forwards PPS as GPIO but does not parse xPPSOffset. |
| Acceleration block | not identified in configured stream | no | none | none | No configured SBF acceleration aiding source was confirmed. |

| Question | Answer |
| --- | --- |
| What does SBF latitude/longitude return? | PVTGeodetic returns latitude and longitude as `f8` radians. |

### Why NMEA and SBF may not look identical

Normal differences:

- NMEA raw latitude/longitude are text degree-minute format; SBF uses binary radians.
- NMEA GGA altitude is MSL altitude; SBF PVTGeodetic height is ellipsoid height.
- NMEA lacks GPS week/TOW in the recorded sentences; SBF has TOW/WNc.
- NMEA USB and SBF UART can have different latency and host receive timestamps.
- NMEA is rounded/formatted text; SBF is binary and higher-detail.
- VTG/RMC course over ground can be blank at low speed; HDT heading can still be valid.

Bug-indicating differences:

- GPS HDT heading is valid but Teensy heading is invalid (`ovf`).
- Saved CV7 config disables external heading even though heading aiding is needed.
- Any raw NMEA GGA vs Teensy SBF position difference after converting height/time/frame would be a bug; latest data does not show this.

Timing error formula:

`position_error_m = speed_mps * time_error_s`

| Speed | 0.01 s error | 0.1 s error | 1.0 s error |
| ---: | ---: | ---: | ---: |
| 1 m/s | 0.01 m | 0.1 m | 1 m |
| 5 m/s | 0.05 m | 0.5 m | 5 m |
| 15 m/s | 0.15 m | 1.5 m | 15 m |
| 30 m/s | 0.3 m | 3 m | 30 m |

A 0.1 s timing error cannot alone explain hundred-meter differences at normal vehicle speeds; a multi-second error, wrong frame/heading, or wrong data source would be needed.
